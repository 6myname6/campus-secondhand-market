# 二手交易平台 — 四人小组分工文档

## 项目概述

本项目是一个 **校园二手交易平台**，采用 C/S 架构：
- **服务端**：Java Netty + MySQL + HikariCP，基于 JSON-over-TCP 协议
- **客户端**：Java Swing 桌面应用，提供完整图形界面
- 支持用户注册登录、商品发布浏览、收藏、即时聊天、订单全生命周期管理

## 组员B：服务端网络层与认证 (Server Network & Auth)

### 职责说明
负责服务端的"门户"：Netty 网络通信框架搭建、JSON 请求解析与路由分发、参数校验、用户认证体系（Token 管理）、用户注册登录业务。

### 负责的文件清单

#### 1. Netty 通信核心
| 文件 | 所在路径 | 说明 |
|------|----------|------|
| `NettyServer.java` | `server/.../netty/` | **服务端入口**。启动 Netty 服务器监听 8081 端口，配置 LineBasedFrameDecoder（最大10MB）、UTF-8 编解码器、boss/worker 事件循环组。ServerHandler 通过 DefaultEventExecutorGroup（16线程）在业务线程池中执行，避免阻塞 I/O 线程 |
| `ServerHandler.java` | `server/.../netty/` | **请求分发核心**。解析 JSON → Request 对象，根据 `action` 字段路由到对应服务方法，维护 `dispatchMap`（`Map<String, Function<JsonObject, Response>>`），实现 `requireAuth` Token 校验包装器 |
| `Request.java` | `server/.../netty/` | 请求 DTO：`{ action, params, _reqId }` |
| `Response.java` | `server/.../netty/` | 响应 DTO：`{ success, data, message }` |
| `ParamValidator.java` | `server/.../netty/` | 参数校验工具：requireInt、requireStr、requirePositiveInt、requireBigDecimal、requireStringLength 等 |

#### 2. 认证服务
| 文件 | 所在路径 | 说明 |
|------|----------|------|
| `AuthService.java` | `server/.../service/` | **Token 认证**。基于 ConcurrentHashMap 的内存 Token 管理：login 生成 UUID Token（后登录挤掉前登录）、validate 校验并刷新有效期、logout 注销。Token 30分钟无活动自动过期，ScheduledExecutorService 定时清理 |

#### 3. 用户服务
| 文件 | 所在路径 | 说明 |
|------|----------|------|
| `UserService.java` | `server/.../service/` | 用户管理：注册（重名检查 + BCrypt 哈希密码）、登录（BCrypt 校验 + 状态检查 + 5秒3次限流）、findByUsername、findById、updateStatus（启用/禁用）、updateProfile（电话/邮箱/头像）、listUsers（分页）、countUsers |

#### 4. 服务基类
| 文件 | 所在路径 | 说明 |
|------|----------|------|
| `BaseService.java` | `server/.../service/` | 抽象基类。提供事务管理方法 `execTx`（带返回值）和 `execTxVoid`（无返回值），支持事务传播（ThreadLocal 复用连接，嵌套调用共享事务），自动处理提交/回滚 |

#### 5. 测试
| 文件 | 所在路径 | 说明 |
|------|----------|------|
| `CrudTest.java` | `server/.../` | 所有实体的 CRUD 冒烟测试（事务回滚，不污染数据库） |
| `UserServiceTest.java` | `server/.../service/` | UserService 单元测试：注册登录、查询、状态更新、分页 |

### 关键技术点
- Netty 的 `LineBasedFrameDecoder` + `StringDecoder/Encoder` 实现基于换行符的 TCP 消息协议
- `DefaultEventExecutorGroup` 业务线程池，将数据库操作与 I/O 线程分离，避免阻塞
- 函数式路由分发（`Map<String, Function>` + Lambda），50 个 action 覆盖全部模块
- `requireAuth` 高阶函数包装器：提取 token → 校验 → 注入 `_userId` → 执行业务，业务层通过 `getAuthUserId()` 获取当前用户防越权
- `ConcurrentHashMap` 线程安全的 Token 存储，30 分钟 TTL 自动过期
- `ScheduledExecutorService` 定时清理过期 Token
- BCrypt 密码哈希存储与校验，登录限流（5秒3次）
- 事务传播机制（ThreadLocal），嵌套调用共享同一事务连接

### 对外接口（供组员C接入）


### 关键技术点
- 事务管理：调用 `BaseService.execTx` 确保多表操作的原子性（如创建订单同时插入订单+明细+更新商品状态）

- 分页查询：基于 MySQL LIMIT/OFFSET，返回 total 总数
- 模糊搜索：`LIKE %keyword%` 参数化查询
- 文件路径规范化防路径穿越攻击

## 协作规范

### 代码风格
- Java 17，UTF-8 编码
- 驼峰命名：类名大驼峰、方法/字段小驼峰
- 数据库列名蛇形命名（由 JdbcUtils 自动转换）
- SQL 参数化查询（禁止字符串拼接 SQL）

### 开发流程
1. **组长A** 先完成实体类 + JdbcUtils + init.sql，其他组员以此为基础
2. **组员B** 完成 Netty 框架 + BaseService + AuthService，定义好 action 路由规范
3. **组员C** 在组员B的框架上实现各业务 Service，逐一注册到 dispatchMap
4. **组员D** 先实现 ApiClient + AppContext，再逐个实现视图页面，边开发边联调

### 接口约定
- 服务端 action 命名：`模块.操作`（如 `goods.publish`、`order.create`）
- 请求格式：`{ "action": "xxx", "params": {...}, "_reqId": N }`
- 响应格式：`{ "success": true/false, "data": {...}, "message": "..." }`
- 需认证的操作：params 中携带 `token` 字段
