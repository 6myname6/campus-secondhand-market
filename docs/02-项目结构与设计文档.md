## 3. 通信协议设计

### 3.1 消息格式

**请求 (Request)：**
```json
{
    "action": "goods.findById",
    "params": {
        "goodsId": 1,
        "token": "uuid-token-if-auth-required"
    },
    "_reqId": 1
}
```

**响应 (Response)：**
```json
{
    "success": true,
    "data": { /* 业务数据 */ },
    "message": "操作成功"
}
```

**错误响应：**
```json
{
    "success": false,
    "data": null,
    "message": "商品不存在"
}
```

## 5. 核心流程设计

### 5.1 用户认证流程
```
客户端                       服务端
  │                            │
  │── login(username, pwd) ──►│ UserService.login()
  │                            │   ├─ 查 users 表
  │                            │   ├─ 校验密码
  │                            │   ├─ 校验状态(active)
  │                            │   └─ 返回 User
  │                            │ AuthService.login(userId)
  │                            │   ├─ 生成 UUID Token
  │                            │   └─ 存入 ConcurrentHashMap
  │◄── { token, userId } ─────│
  │                            │
  │── 后续请求(带 token) ────►│ requireAuth() 包装器
  │                            │   ├─ AuthService.validate(token)
  │                            │   ├─ 通过 → 注入 userId → 执行业务
  │                            │   └─ 失败 → 返回 "请先登录"
  │◄── 业务数据 ──────────────│
```

### 5.2 商品发布流程
```
客户端                       服务端
  │                            │
  │── file.upload(base64[]) ──►│ FileService.upload()
  │                            │   └─ 解码 → 写 uploads/ → 返回 URL[]
  │◄── imageUrls[] ───────────│
  │                            │
  │── goods.publish(商品信息) ─►│ GoodsService.publish()
  │                            │   └─ insert goods
  │◄── goodsId ───────────────│
  │                            │
  │── goodsImage.addBatch() ──►│ GoodsImageService.addBatch()
  │  (goodsId + imageUrls)    │   └─ insert goods_images[]
  │◄── success ───────────────│
```

### 5.3 创建订单流程（事务）
```
OrderService.createOrder()
  │
  ├─ 1. 查询商品信息（价格、卖家）
  ├─ 2. 计算订单总金额 = Σ(price × quantity)
  ├─ 3. 生成唯一订单号（时间戳+随机数）
  ├─ 4. INSERT INTO orders (...)
  ├─ 5. 批量 INSERT INTO order_items (...)
  ├─ 6. UPDATE goods SET status='sold' WHERE goods_id IN (...)
  │
  └─ 以上全部在一个事务中 → 任一失败则回滚
```

### 5.4 聊天消息流程
```
发送方客户端                  服务端                      接收方客户端
  │                            │                            │
  │── message.send() ────────►│ MessageService.send()       │
  │                            │  ├─ INSERT message         │
  │                            │  └─ UPDATE conversation    │
  │                            │     (last_message,         │
  │                            │      unread_count+1)       │
  │◄── success ───────────────│                            │
  │                            │                            │
  │                            │  ◄── message.listByConv()  │ (3秒轮询)
  │                            │──── messages[] ──────────►│
```

## 6. 关键设计决策

### 6.1 为什么用 Netty 而不是 HTTP/REST？
- 学习目的：深入理解 TCP 通信和自定义协议
- 低延迟：适合聊天等实时通信场景
- 单连接复用：客户端维持一条 TCP 连接完成所有操作，减少握手开销

### 6.2 为什么用 JSON-over-TCP 而不是 Protobuf？
- JSON 可读性好，便于调试
- Gson 序列化简单，与 Java 对象映射方便
- 项目规模小，JSON 的性能开销可接受

### 6.3 为什么用 ConcurrentHashMap 存 Token 而不是 Redis？
- 项目定位为学习项目，减少外部依赖
- 服务端单实例部署，无分布式需求
- 实现简单，重启即清空（安全性可接受）

### 6.4 为什么要自己写 ORM 而不是用 MyBatis/JPA？
- 学习目的：理解 ORM 底层原理（反射、注解、SQL 生成）
- 项目规模小（8张表），自定义 ORM 足够
- 避免引入重量级框架，降低学习成本

### 6.5 为什么要分模块（server/client）？
- server 模块包含实体类，client 依赖 server 模块复用实体
- 打包时 server 和 client 各自独立为 fat JAR
- 部署时服务端和客户端可运行在不同机器上
